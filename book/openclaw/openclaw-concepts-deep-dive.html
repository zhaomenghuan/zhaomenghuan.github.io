OpenClaw 核心概念深度解析 — Sessions、Channels、Agents 及其关系 ​

作者：探探（researcher）| 日期：2026-03-06 基于 /app/docs/ 官方文档调研整理

一、三大核心概念深度解析 ​

1. Channels（通道） ​

1.1 定义和作用 ​

Channel 是 OpenClaw 与外部消息平台之间的传输层。每个 Channel 对应一个聊天平台的连接，负责接收用户消息并将 Agent 的回复发送回去。

原文引用（channels/index.md）： "OpenClaw can talk to you on any chat app you already use. Each channel connects via the Gateway."

关键特性：路由是确定性的——模型不会选择通道，路由完全由配置决定。

原文引用（channels/channel-routing.md）： "The model does not choose a channel; routing is deterministic and controlled by the host configuration."

1.2 支持的通道类型 ​

OpenClaw 支持的通道覆盖主流聊天平台：

通道标识接入方式备注
WhatsAppwhatsappBaileys + QR 配对最常用，需 QR 扫码
TelegramtelegramBot API (grammY)最快上手（只需 Bot Token）
DiscorddiscordBot API + Gateway支持 Server、Channel、DM

通道可以同时运行：配置多个通道后，OpenClaw 会根据 Bindings 自动路由到对应的 Agent。

1.3 AccountId 机制 ​

一个通道可以有多个账号实例，通过 accountId 区分。

典型场景：同一个 WhatsApp 通道下绑定两个号码（personal 和 biz），分别路由到不同 Agent。

json5
channels: {
  whatsapp: {
    accounts: {
      personal: { /* 个人号配置 */ },
      biz: { /* 工作号配置 */ }
    }
  }
}

原文引用（concepts/multi-agent.md）： "Channels that support multiple accounts (e.g. WhatsApp) use accountId to identify each login."

1.4 DM vs Group 的区别 ​

维度DM（私聊）Group（群聊）
Session Key合并到主 session agent:&lt;id&gt;:&lt;mainKey&gt;独立 agent:&lt;id&gt;:&lt;channel&gt;:group:&lt;groupId&gt;
访问控制dmPolicy 控制groupPolicy 控制
触发方式直接发消息即触发通常需要 @提及
Heartbeat会执行跳过

1.5 DM Policy 和 Group Policy ​

DM Policy（私聊策略）：

策略说明
pairing默认。未知发送者需要短码配对审批后才能对话
allowlist只有 allowFrom 列表中的发送者可以对话
open所有人都可以对话（不推荐）

Group Policy（群聊策略）：

策略说明
allowlist默认。只有配置了的群可以响应
open所有群都可以响应
disabled禁用所有群聊响应

原文引用（channels/pairing.md）： "When a channel is configured with DM policy pairing, unknown senders get a short code and their message is not processed until you approve."

配对码特性：8 位大写字母，1 小时过期，每通道最多 3 个待审请求。

1.6 Channel 的认证和登录 ​

不同通道的认证方式：

  • WhatsApp：QR 扫码配对，凭证存在 ~/.openclaw/credentials/whatsapp/&lt;accountId&gt;/
  • Telegram：Bot Token（通过 BotFather 获取）
  • Discord：Bot Token（需开启 Message Content Intent）
  • 飞书/大象：应用凭证（App ID + Secret）
  • Signal：signal-cli 注册
bash
# WhatsApp 登录示例
openclaw channels login --channel whatsapp --account personal

2. Agents（智能体） ​

2.1 完整定义 ​

Agent 是 OpenClaw 中一个完全隔离的"大脑"，拥有三大独立组件：

原文引用（concepts/multi-agent.md）： "An agent is a fully scoped brain with its own: Workspace, State directory (agentDir), Session store."

组件路径作用
Workspace~/.openclaw/workspace-&lt;agentId&gt;人格文件、记忆、Skills
agentDir~/.openclaw/agents/&lt;agentId&gt;/agent认证信息、模型注册表
Session Store~/.openclaw/agents/&lt;agentId&gt;/sessions聊天历史、路由状态

2.2 单 Agent 模式 vs 多 Agent 模式 ​

单 Agent 模式（默认）：

  • agentId 默认为 main
  • Session Key 为 agent:main:main
  • Workspace 默认 ~/.openclaw/workspace

原文引用（concepts/multi-agent.md）： "If you do nothing, OpenClaw runs a single agent. agentId defaults to main."

多 Agent 模式：

  • 在 agents.list 中定义多个 Agent
  • 通过 Bindings 路由消息到不同 Agent
  • 每个 Agent 是完全独立的人格
json5
agents: {
  list: [
    { id: "main", default: true, name: "夏夏", workspace: "~/.openclaw/workspace" },
    { id: "coder", name: "码哥", workspace: "~/.openclaw/workspace-coder" },
    { id: "researcher", name: "探探", workspace: "~/.openclaw/workspace-researcher" }
  ]
}

2.3 Agent 的隔离性 ​

三重隔离：

① 认证隔离：

原文引用（concepts/multi-agent.md）： "Auth profiles are per-agent. Each agent reads from its own auth-profiles.json. Main agent credentials are not shared automatically. Never reuse agentDir across agents (it causes auth/session collisions)."

② 会话隔离：每个 Agent 的 Session Store 独立，agent:main:... 和 agent:researcher:... 完全分开。

③ 文件隔离：每个 Agent 的 Workspace 独立，SOUL.md、AGENTS.md、memory/ 等互不可见（除非通过 memorySearch.extraPaths 显式共享）。

2.4 Agent 配置字段 ​

json5
{
  id: "researcher",          // 唯一标识（路由和 bindings 用）
  default: false,            // 是否为默认 Agent
  name: "探探",              // 显示名称
  workspace: "~/.openclaw/workspace-researcher",  // 工作区路径
  agentDir: "~/.openclaw/agents/researcher/agent", // 状态目录
  model: "anthropic/claude-opus-4-5",  // 可选，专用模型
  identity: { name: "探探" },  // 身份信息（群聊显示名）
  sandbox: {                  // 可选，沙箱配置
    mode: "all",
    scope: "agent"
  },
  tools: {                    // 可选，工具限制
    allow: ["read", "exec"],
    deny: ["write"]
  },
  groupChat: {                // 可选，群聊配置
    mentionPatterns: ["@探探"]
  }
}

2.5 Per-Agent Sandbox 和 Tool 配置 ​

从 v2026.1.6 开始，每个 Agent 可以有独立的沙箱和工具策略：

原文引用（concepts/multi-agent.md）： "Starting with v2026.1.6, each agent can have its own sandbox and tool restrictions."

json5
{
  id: "family",
  sandbox: { mode: "all", scope: "agent" },  // 始终沙箱化
  tools: {
    allow: ["read"],           // 只允许 read
    deny: ["exec", "write"]    // 禁止 exec 和 write
  }
}

注意：tools.elevated 是全局的，不支持 per-agent 配置。

2.6 Skills 的 per-agent vs shared 机制 ​

Skills 从三个位置加载（名称冲突时，工作区优先）：

  1. Bundled（内置）：随安装包发布
  2. Managed/Local：~/.openclaw/skills（全局共享）
  3. Workspace：&lt;workspace&gt;/skills（Agent 专属）

原文引用（concepts/agent.md）： "OpenClaw loads skills from three locations (workspace wins on name conflict): Bundled, Managed/local: ~/.openclaw/skills, Workspace: <workspace>/skills"

3. Sessions（会话） ​

3.1 定义和生命周期 ​

Session 是 Agent 与用户之间一段对话的上下文容器。它存储聊天历史、模型状态和路由信息。

生命周期：

  • 创建：当有消息到达且没有匹配的活跃 Session 时自动创建
  • 复用：Session 在到期前一直复用
  • 过期检查：在下一条消息到达时评估
  • 每日重置：默认每天凌晨 4:00（Gateway 本地时间）
  • 空闲重置（可选）：idleMinutes 设置滑动窗口
  • 手动重置：发送 /new 或 /reset

原文引用（concepts/session.md）： "Reset policy: sessions are reused until they expire, and expiry is evaluated on the next inbound message. Daily reset: defaults to 4:00 AM local time on the gateway host."

3.2 Session Key 的格式 ​

Session Key 是 Session 的唯一标识，格式取决于消息类型：

DM（私聊）：

dmScope 设置Key 格式
main（默认）agent:&lt;agentId&gt;:&lt;mainKey&gt;
per-peeragent:&lt;agentId&gt;:dm:&lt;peerId&gt;
per-channel-peeragent:&lt;agentId&gt;:&lt;channel&gt;:dm:&lt;peerId&gt;
per-account-channel-peeragent:&lt;agentId&gt;:&lt;channel&gt;:&lt;accountId&gt;:dm:&lt;peerId&gt;

Group（群聊）：

  • 群组：agent:&lt;agentId&gt;:&lt;channel&gt;:group:&lt;id&gt;
  • 频道/房间：agent:&lt;agentId&gt;:&lt;channel&gt;:channel:&lt;id&gt;
  • Telegram 论坛主题：agent:&lt;agentId&gt;:telegram:group:&lt;groupId&gt;:topic:&lt;topicId&gt;
  • Discord/Slack 线程：追加 :thread:&lt;threadId&gt;

其他来源：

  • Cron 任务：cron:&lt;job.id&gt;
  • Webhook：hook:&lt;uuid&gt;
  • Node：node-&lt;nodeId&gt;
  • Sub-agent：agent:&lt;agentId&gt;:subagent:&lt;uuid&gt;

原文引用（channels/channel-routing.md）： "Direct messages collapse to the agent's main session: agent:<agentId>:<mainKey>"

3.3 mainKey 的概念 ​

默认情况下（dmScope: "main"），所有 DM 都合并到 Agent 的主 Session：agent:&lt;agentId&gt;:main。

这意味着：来自 WhatsApp、Telegram、飞书等不同通道的私聊，都会进入同一个会话上下文。

原文引用（concepts/session.md）： "main (default): all DMs share the main session for continuity."

mainKey 可以自定义：

json5
session: { mainKey: "main" }  // 默认值

⚠️ 安全警告：如果你的 Agent 接收多人 DM（如开放了 dmPolicy: "open"），默认的 main 模式会导致不同用户共享上下文，存在信息泄露风险。建议改为 per-channel-peer。

3.4 Session 的可见性 ​

通过 tools.sessions.visibility 控制 Agent 能看到哪些 Session：

值说明
self只看自己当前的 Session
tree（默认）自己 + 自己生成的 Sub-agent Session
agent同一 agentId 下的所有 Session
all所有 Agent 的所有 Session

原文引用（concepts/session-tool.md）： "tools.sessions.visibility defaults to tree (current session + spawned subagent sessions)."

沙箱内的 Session 还受 agents.defaults.sandbox.sessionToolsVisibility 限制（默认 spawned，会强制 clamp 到 tree）。

3.5 Session Tools ​

OpenClaw 提供四个 Session 工具：

① sessions_list：列出可见的 Session

  • 支持 kinds 过滤（main/group/cron/hook/node/other）
  • 支持 activeMinutes 只看最近活跃的
  • 支持 messageLimit 获取最近消息

② sessions_history：获取指定 Session 的聊天记录

  • 需要 sessionKey
  • 支持 includeTools 参数
  • 支持 limit 限制条数

③ sessions_send：向另一个 Session 发送消息

  • 需要 sessionKey + message
  • timeoutSeconds > 0：等待回复
  • timeoutSeconds = 0：fire-and-forget
  • 支持 ping-pong 来回对话（最多 5 轮，回复 REPLY_SKIP 终止）
  • 结束后执行 announce 步骤（回复 ANNOUNCE_SKIP 跳过）

④ sessions_spawn：生成子 Agent 运行

  • 创建 agent:&lt;agentId&gt;:subagent:&lt;uuid&gt; Session
  • 非阻塞，立即返回
  • 完成后通过 announce 向请求者通道推送结果
  • 支持 thread: true 绑定到消息线程

3.6 Sub-agent Sessions vs Inter-agent Sessions ​

维度Sub-agent SessionInter-agent Session
创建方式sessions_spawnsessions_send
Session Keyagent:&lt;id&gt;:subagent:&lt;uuid&gt;目标 Agent 已有的 Session
关系父子关系平级关系
工具权限默认无 Session Tools完整工具
生命周期完成后自动归档（默认 60 分钟）持久存在
配置要求无额外配置需开启 agentToAgent
结果传递announce 推送到请求者通道直接回复 + 可选 announce

原文引用（tools/subagents.md）： "Sub-agents are background agent runs spawned from an existing agent run. They run in their own session and, when finished, announce their result back to the requester chat channel."

二、三者之间的关系 ​

1. Channel → Agent 的路由机制（Bindings） ​

Bindings 是消息路由的核心——决定"哪个通道的消息由哪个 Agent 处理"。

路由优先级 ​

路由遵循最具体匹配优先原则：

1. peer 精确匹配（指定 DM/群聊 ID）        → 最高优先级
2. parentPeer 匹配（线程继承）
3. guildId + roles 匹配（Discord 角色路由）
4. guildId 匹配（Discord 服务器）
5. teamId 匹配（Slack 团队）
6. accountId 匹配（指定账号）
7. channel 匹配（通道级别通配）
8. default Agent 兜底                      → 最低优先级

原文引用（concepts/multi-agent.md）： "If multiple bindings match in the same tier, the first one in config order wins."

多 Binding 冲突解决 ​

  • 同一层级中，配置顺序靠前的优先
  • 一个 binding 设置多个 match 字段时，所有字段必须同时匹配（AND 语义）
  • 省略 accountId 的 binding 只匹配默认账号
  • accountId: "*" 表示匹配所有账号
json5
bindings: [
  // 精确：特定群聊走 coder（最先匹配）
  { agentId: "coder", match: { channel: "daxiang", peer: { kind: "group", id: "GROUP_ID" } } },
  // 通配：大象其他消息走 main
  { agentId: "main", match: { channel: "daxiang" } },
  // 飞书走 researcher
  { agentId: "researcher", match: { channel: "feishu" } }
]

2. Agent → Session 的映射关系 ​

一个 Agent 多个 Sessions ​

一个 Agent 可以同时拥有多个 Session：

Agent "main"（夏夏）
├── agent:main:main              ← DM 主 Session（所有私聊汇聚）
├── agent:main:daxiang:group:123 ← 大象群聊 Session
├── agent:main:webchat:dm:user1  ← WebChat DM
└── agent:main:subagent:uuid-xxx ← Sub-agent Session

Session Key 的生成规则 ​

Session Key 由以下因素共同决定：

agentId + dmScope 设置 + channel + 消息类型(DM/Group) + peerId
  • DM 默认合并到 agent:&lt;agentId&gt;:main
  • 群聊始终独立 agent:&lt;agentId&gt;:&lt;channel&gt;:group:&lt;groupId&gt;
  • Sub-agent 用 UUID agent:&lt;agentId&gt;:subagent:&lt;uuid&gt;

3. Channel → Session 的关系 ​

Channel + Peer 共同决定 Session Key：

消息到达                         Session Key
─────────────────────────────────────────────────────────
大象 DM from user1           → agent:main:main (dmScope=main)
飞书 DM from user1           → agent:researcher:main (dmScope=main)
大象群聊 group123             → agent:main:daxiang:group:123
Telegram group -100xxx       → agent:main:telegram:group:-100xxx
Discord thread in channel    → agent:main:discord:channel:123:thread:456

4. Agent 间通信（agentToAgent）的 Session 机制 ​

Agent 间通信通过 sessions_send 实现，需要显式开启：

json5
tools: {
  agentToAgent: {
    enabled: true,
    allow: ["main", "coder", "researcher"]
  },
  sessions: { visibility: "all" }
}

通信流程：

  1. Agent A 调用 sessions_list 查看所有 Session
  2. Agent A 找到 Agent B 的 Session Key
  3. Agent A 调用 sessions_send(sessionKey, message, timeoutSeconds)
  4. Agent B 的 Session 收到消息，自动处理并回复
  5. 支持 ping-pong 来回（最多 maxPingPongTurns 轮，默认 5）
  6. 回复 REPLY_SKIP 终止循环
  7. 循环结束后，执行 announce 步骤（目标 Agent 可选向其通道用户推送）

原文引用（concepts/session-tool.md）： "After the primary run completes, OpenClaw runs a reply-back loop: Round 2+ alternates between requester and target agents. Reply exactly REPLY_SKIP to stop the ping‑pong. Max turns is session.agentToAgent.maxPingPongTurns (0–5, default 5)."

         │  ③ 推送到夏夏的通道（大象）  │
         │     包含:                   │
         │     - Status: completed     │
         │     - Result: 报告内容      │
         │     - 运行时间、Token 用量   │
         └────────────────────────────┘

                  │
                  ▼
         夏夏（或欢哥）在大象收到结果通知

         60 分钟后 → Sub-agent Session 自动归档
